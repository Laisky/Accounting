# Internationalization (i18n) for the Web Frontend

This document describes the i18n strategy for the `web/` frontend (React 19 + Vite).
The design mirrors the Laisky `one-api` frontend so the two projects stay consistent.

## 1. Architecture Overview

We use **[i18next](https://www.i18next.com/)**, **[react-i18next](https://react.i18next.com/)**,
and **[i18next-browser-languagedetector](https://github.com/i18next/i18next-browser-languageDetector)**.

- **Supported languages**: English (`en`, source of truth), Simplified Chinese (`zh`),
  French (`fr`), Spanish (`es`), Japanese (`ja`).
- **Storage**: the chosen language is persisted in `localStorage` under `i18nextLng`.
- **Fallback order**: saved preference → browser language → English.
- **Loading strategy**: English is bundled synchronously so the first render is immediate and
  server-side/tests never flash untranslated keys. Every other language is a **lazily imported
  chunk** loaded on demand when the user switches to it (`loadLanguageResources` in `src/i18n/index.ts`).
- **`<html lang>`** is kept in sync with the active language.

## 2. File Structure

```text
src/
├── i18n/
│   ├── index.ts                 # i18next config, detector, lazy loaders, changeAppLanguage()
│   └── locales/
│       ├── en/                  # canonical source
│       │   ├── index.ts         # merges the namespaces into one `translation` bundle
│       │   ├── common.json      # shared strings, nav, flow words, status/error
│       │   ├── auth.json        # authentication + login-method labels
│       │   ├── ledger.json      # desktop ledger console
│       │   ├── imports.json     # Wacai import workspace
│       │   ├── reports.json     # reporting drilldowns
│       │   └── mobile.json      # Wacai-style mobile shell (Accounts/Record/Reports/Me)
│       ├── zh/ …                # same filenames, translated values
│       ├── fr/ …
│       ├── es/ …
│       └── ja/ …
├── components/
│   └── LanguageSelector.tsx     # accessible native-select language switcher
└── vite-env.d.ts                # provides `import.meta.env` typing
```

Each namespace JSON is keyed by its namespace name (e.g. `common.json` → `{ "common": { … } }`),
and `locales/<lng>/index.ts` spreads them into a single flat `translation` bundle. Usage is
therefore `t('common.signOut')`, `t('auth.heading')`, `t('reports.tabs.category')`, etc.

## 3. Usage in Components

```tsx
import { useTranslation } from 'react-i18next';

export function Example() {
  const { t } = useTranslation();
  return <button>{t('common.postEntry')}</button>;
}
```

- **Interpolation** uses `{{name}}` placeholders: `t('mobile.budget.total', { amount })`.
- Numeric counts are passed as a neutral `{{value}}` variable (not the i18next-reserved
  `count`) so no accidental pluralization occurs: `t('common.entriesCount', { value })`.
- Module-level helpers that need translations (e.g. `reportHeading`, `submitLabel`) receive the
  `TFunction` as their first parameter instead of calling a hook.
- Switch languages with `changeAppLanguage(code)` from `src/i18n`, which loads the target chunk
  before calling `i18n.changeLanguage`.

## 4. Language Switcher

`LanguageSelector` (rendered on the mobile **Me** tab) is a native `<select>` for accessibility,
listing each language in its own script (English / 简体中文 / Français / Español / 日本語).

## 5. Alignment Check (CI gate)

`scripts/check-i18n-locales.mjs` (run via `npm run check:i18n`, wired into `make frontend-lint`)
enforces that every language has the **same files, same key paths, and same line counts** as the
English reference. A missing key, an extra key, or a reflowed file fails the build. This keeps all
five languages structurally identical to the source.

## 6. Adding Strings

1. Add the key/value to the relevant `en/*.json` namespace (English is canonical).
2. Add the **same key** to the other four languages with a translated value, keeping the file
   structure and line count identical.
3. Reference it in code via `t('namespace.key')`.
4. Run `npm run check:i18n` to confirm alignment.

## 7. Adding a Language

1. Create `src/i18n/locales/<lng>/` with all namespace JSON files plus an `index.ts` barrel
   (copy an existing language's `index.ts`).
2. Register the code in `supportedLanguages` and add a lazy loader in `localeLoaders`
   (`src/i18n/index.ts`).
3. Add the language to `languages` in `src/components/LanguageSelector.tsx`.
4. Add the code to `supportedLanguages` in `scripts/check-i18n-locales.mjs` and run the check.

## 8. Known Limitations

Date and currency values use `Intl` with a fixed `en-US` locale for stable, test-friendly output.
Only user-facing copy is translated; numeric/date formatting localization is intentionally out of
scope for this pass.
