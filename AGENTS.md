# Agent Instructions

- **Read private instructions FIRST:** Whenever `.github/instructions/laisky.instructions.md` exists, you MUST read and follow it before any other work; it holds sensitive environment-specific details (hosts, credentials, private tooling) and its contents must never be leaked, quoted, logged, or committed.
- **Project:** Accounting is a browser-first bookkeeping system for personal, family, shared-living, trip, project, and small-team ledgers; prioritize correctness, data integrity, auditability, and user data ownership over raw feature velocity. It is a single repository with three parts: `backend/` (Go API server) and `cli/` (Go) are Go modules; `web/` is a React 19 + TypeScript 6 + Vite 8 frontend managed with pnpm. In development `make dev` runs the Vite dev server (`:5173`) proxying `/api` to the Go backend (`:8080`); in production one Go binary serves the JSON API, built `web/dist` assets, and the SPA fallback.
- **Source of truth:** Keep code, comments, scripts, and docs aligned with `docs/arch/arch.md` (architecture, product model, JSON API) and `docs/arch/i18n.md` (localization), and with shipped backend/CLI/frontend behavior; if a requested change conflicts with them, resolve the conflict explicitly instead of inventing a third behavior, and update the relevant doc in the same change.
- **English only:** Output **English** for all code, comments, chat, documents, logs, and UI copy, regardless of input language; user-facing UI strings must go through the i18n layer, never hard-coded.
- **Secrets:** Treat any API keys, tokens, or secrets in the repo or instructions as sensitive — never echo, log, commit, or expose them. Auth spans email/password, TOTP, WebAuthn passkeys, and external GraphQL-based SSO, so APIs and logs must never return or record password hashes, session tokens, TOTP secrets, passkey/WebAuthn key material, external SSO bearer tokens, or email verification and password-reset codes.
- **Research over memory:** Do NOT trust model memory for tools, platforms, APIs, or best practices — verify against current (2026) sources before implementing; capture durable, reusable findings in the relevant `docs/arch` note (or a new `docs/` reference) so later work can cite them instead of re-researching.
- **Testing & CI:** After any change, run the gates from the repo root: `make lint` and `make test` (backend `-race -cover`, CLI `-race -cover`, and frontend Vitest); run `make e2e` (Playwright) when behavior spanning API and UI changes. Narrower checks: `cd backend && go test -race -cover ./...`, `cd cli && go test -race -cover ./...`, `pnpm --dir web run test`, `pnpm --dir web run lint`, `pnpm --dir web run check:i18n`. Never skip these gates; add or update tests for every feature and bug fix instead of one-off scripts, use `github.com/stretchr/testify/require` for Go assertions, and continuously improve coverage.
- **Acceptance beyond existing tests:** Do NOT treat the current test suite or current behavior as the acceptance baseline — derive the REAL intended behavior from the overall implementation and from `docs/arch/arch.md`, then build a comprehensive test set (behavior tests plus an API/UI e2e matrix) against that intent to surface genuine defects; fix and re-verify them before declaring acceptance.
- **Go style:** Target **Go 1.26.4** with modern syntax in both `backend/` and `cli/`; no manually written file may exceed **800 lines** (generated files exempt); thread `context.Context` through call chains for lifecycle and cancellation; every exported function, method, and interface must have a comment starting with its name describing purpose, parameters, and return values in complete sentences.
- **Errors:** Use `github.com/Laisky/errors/v2` — never return bare errors, wrap with `errors.Wrap`/`errors.Wrapf`/`errors.WithStack`; handle errors as close to the source as possible; never swallow or shadow them; process each error exactly once — either **returned** or **logged**, never both.
- **Logging:** Use structured `github.com/Laisky/zap` logging with typed fields such as `zap.Error(err)` (no `fmt.Sprintf` messages); on Gin request paths pull the context-aware logger with `gmw.GetLogger(c)` **once per function** and store it locally rather than using a global logger; add targeted DEBUG logs to aid diagnosis and keep the useful ones; **never** log secrets.
- **Security:** Use constant-time comparisons (`crypto/subtle`, e.g. `subtle.ConstantTimeCompare`) for tokens, signatures, and hashes; keep password hashing at the OWASP PBKDF2-SHA256 recommendation (the codebase uses **600,000** iterations — do not lower it); keep session cookies HttpOnly and Secure in production; never use untrusted input directly in storage queries or memory allocation — always validate and sanitize to prevent injection and DoS.
- **Persistence & time:** Route all storage through the `backend/internal/persistence` boundary (memory and JSON-file drivers today, selected via `ACCOUNTING_PERSISTENCE_DRIVER`); do not bypass it or assume a specific database. Use **UTC** for all server, storage, and API time handling; date-range queries include the entire final day (end just before 00:00 UTC of the next day).
- **Frontend:** Work in `web/` with pnpm via `pnpm --dir web <cmd>` (pinned `packageManager`); it is a mobile-first React 19 + TypeScript 6 + Vite 8 app. Keep ESLint clean, keep the TypeScript build (`tsc -b`) passing, cover logic with Vitest + Testing Library and cross-cutting flows with Playwright, and keep every i18n locale bundle in sync so `check:i18n` passes; add user-facing strings through i18next, never hard-code them.
- **Concurrency & tracking:** Multiple agents may edit code concurrently — **preserve others' changes**, avoid overwriting, halt and notify only on an irreconcilable conflict; use the TODOs tool to track tasks and keep it current and actionable.
- **Method:** Plan before acting — check constraints and prerequisites, order operations correctly, prefer acting with available info over blocking on optional details (actions are final once executed); debug by generating and prioritizing hypotheses, testing the most likely first, and revising the plan when observations disprove one; when an anomaly is found, REPRODUCE it with a behavior test to rule out false positives BEFORE fixing, then accept the fix by turning that same test red→green; retry transient errors up to a reasonable limit and change strategy for non-transient failures; ask the user only when missing info blocks the next step.
- **Sub-agents:** Actively use sub-agents for **parallelism** (dispatch independent subtasks concurrently in a single batch) and **context isolation** (offload searches/sweeps/lookups); each must follow these same instructions, stay scoped to a well-defined subtask, and return only **distilled key information** — never raw tool output or full file dumps.
- **Code search:** Prefer `ast-grep --lang <language> -p '<pattern>'` for syntax-aware code search; use plain-text search only when explicitly requested or ast-grep does not fit.
